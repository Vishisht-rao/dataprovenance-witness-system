# Qualtrics Witness

A cryptographic witness for Qualtrics survey responses. Qualtrics has no JSON
export for researchers that is independently fetched (a researcher could
always edit a downloaded CSV by hand before showing it to anyone). So this
witness pulls each response **directly from Qualtrics** the moment it
arrives, by email, and signs the result with a key the researcher never
holds.

## The three roles

| Role | Who | What they do |
|---|---|---|
| **Researcher** | runs the survey | Builds the Qualtrics survey + a Workflow that emails the witness account on every response. Never touches the keys. |
| **Witness admin** | runs this code | Owns the Gmail inbox and the secret key. Runs `extract_and_sign.py` to ingest, canonicalize, and sign. |
| **Verifier** | anyone (reviewer, reader) | Needs only `witness_pk.ed25519` + the published `_canonical.jsonl` + `_receipt.json`. Runs `verify_data.py`. |

The witness admin should be someone other than the researcher (for example a
lab manager, an IRB office, or a third party service). The guarantee is that
this dataset was witnessed independently, not that the researcher promises
this is real.

## How it works, end to end

1. **Researcher** creates the Qualtrics survey as normal, then adds a
   **Workflow** with a "Survey Response" trigger and a **Send Email** task.
   The email goes to the witness's Gmail address. Qualtrics fills the email
   body with the response's **Single Response Report (SRR)** link via piped
   text. This is just a pointer, not trusted data.
2. Qualtrics sends the email through `qemailserver.com`. Gmail authenticates
   it on arrival and stamps the result in the `Authentication-Results`
   header (DKIM/DMARC pass/fail). The witness checks this header. It does
   not perform the verification itself.
3. **Witness admin** runs:
   ```
   python extract_and_sign.py <SURVEY_ID> --base_url https://your-org.qualtrics.com
   ```
   which runs, in order:
   * `gmail_ingestor.py`: finds unread mail with a subject matching the
     survey ID, keeps only mail that passed DKIM and DMARC from
     `qemailserver.com`, follows the SRR link, fetches the **live SRR page**
     (not the email body), and parses the response ID and answers out of
     it. Writes `outputs/<SURVEY_ID>/<SURVEY_ID>_srr.jsonl`.
   * `email_to_canonical_format.py`: normalizes whitespace, keeps only
     `response_id` and `answers`, and sorts both the answer keys and the
     rows, so the same set of responses always serializes to the same bytes
     regardless of arrival order. Writes `<SURVEY_ID>_canonical.jsonl`.
   * `sign_canonical_email.py`: hashes the canonical file with SHA-256,
     signs the hash with the Ed25519 secret key, and writes
     `<SURVEY_ID>_receipt.json` (`dataset_hash`, `dataset_sig`, `row_count`,
     `columns`).
4. The witness admin publishes the two files
   `<SURVEY_ID>_canonical.jsonl` + `<SURVEY_ID>_receipt.json` (see
   `outputs/` for two real pilot examples).
5. **Verifier** runs:
   ```
   python verify_data.py --data outputs/<SURVEY_ID>/<SURVEY_ID>_canonical.jsonl \
                          --receipt outputs/<SURVEY_ID>/<SURVEY_ID>_receipt.json
   ```
   This recomputes the dataset's hash and checks the signature against
   `witness_pk.ed25519`. `SUCCESS` means every row in this file was
   witnessed, byte for byte, by the holder of the secret key, and nothing
   in it has been edited, added, or removed since.

   > Note: the verifier checks `dataset_sig` against the data. `row_count`
   > and `columns` in the receipt are disclosed for convenience, but they
   > are not themselves part of the signed bytes.

## Setup (witness admin only)

```
pip install -r requirements.txt
python key_generator.py        # writes witness_sk.ed25519 + witness_pk.ed25519
```
`witness_sk.ed25519` must stay on the witness machine only. It is ignored
by git. `witness_pk.ed25519` is meant to be shared (it is what verifiers
use) and is tracked in this repo.

You also need a Gmail API OAuth client for the witness's inbox:
1. In Google Cloud Console, create a project, enable the Gmail API, and
   create an OAuth Desktop client. Download it as `credentials.json` in
   this folder (ignored by git, get your own, do not reuse anyone else's).
2. First run of `gmail_ingestor.py` opens a browser to authorize and saves
   `token.json` (also ignored by git, it is a live credential to the
   inbox).

## What this does and does not prove

* **Proves:** if verification passes, every response in the published file
  is a genuine entry fetched from Qualtrics' own SRR page. It was not made
  up after the fact by the researcher, or generated by AI, and it is byte
  for byte identical to what the witness signed.
* **Does not prove:** that the researcher didn't submit a number of
  responses themselves, or via a bot. The witness only attests that a real
  Qualtrics entry exists, not who or what filled it in. It also does not
  prove completeness. Because the researcher configures the Workflow, they
  could set it to email the witness only for responses matching certain
  criteria (for example only when an answer was A, B, or C, never D), so
  the signed set may be a handpicked subset of everything actually
  collected. The guarantee is tamper evidence of what was signed, not
  completeness or who supplied the answers.

## Other available options

`sign_canonical_email.py` currently signs all responses together as one
dataset with one signature. It can instead be configured to sign each
response individually, and to redact specified columns before signing, if
a study has privacy sensitive answers. See the commented out `sign_rows`
and redaction code in that file.
